## 1. このファイルの目的
このファイルは、本プロジェクトにおける AI エージェントおよび開発メンバー向けの共通運用ルールを定義するためのものである。  
Unity を用いた複数人開発を円滑に進めるため、作業範囲、変更方針、Git 運用、報告方法、言語方針、環境共通ルールをここにまとめる。

このファイルで扱うのは、主に以下である。

- プロジェクト全体の作業ルール
- Git / ブランチ / PR のルール
- Unity 特有の注意点
- AI エージェントの振る舞い
- 日本語で統一する対象
- Windows / Mac 混在開発時の共通ルール

コードの命名規則、記述規約、コメント規約などの詳細は、別ファイル `Docs/01_Overview/CodingRules.md` を参照すること。

---

## 2. 基本方針
- 変更はできるだけ小さな単位で行う。
- 1つの作業ブランチでは、1つの目的に集中する。
- 無関係な修正を同時に行わない。
- 既存の構成、命名、実装方針を尊重する。
- 大きな変更は、勝手に全面置換せず、影響範囲を明確にして進める。
- 仕様が不明確な場合は、README、既存コード、Issue、設計メモを優先して判断する。
- 判断に迷う場合は、保守的な変更を優先する。
- 複数人で同時開発していることを常に意識し、競合や手戻りを最小限にする。
- 変更前後で差分を確認し、意図しない変更が混ざっていないことを確認する。

---

## 3. 言語方針
このプロジェクトでは、人が読む必要のある情報は原則として日本語で記述する。

### 日本語で記述するもの
- タスクの説明
- ウォークスルー
- Issue の説明
- PR の説明
- 作業報告
- コード内コメント
- ドキュメント
- TODO コメント
- デバッグ手順
- 引き継ぎメモ
- レビューコメント
- 作業手順書

### 例外
以下は必要に応じて英語を許容する。

- Unity や C# の予約語
- API 名、ライブラリ名、クラス名、メソッド名
- 一般的な技術用語として英語の方が自然なもの
- ブランチ名
- コミット種別（feat, fix, docs など）
- Unity の標準機能名、Inspector 上の正式名称

### コメント方針
- コメントは日本語で書く。
- 「何をしているか」だけでなく、「なぜ必要か」が分かるコメントを優先する。
- 一時的なメモや曖昧なコメントは残さない。
- 将来読んだ人が誤解しやすいコメントは書かない。

## 5. 作業範囲
### 変更してよいもの
- 依頼された機能追加
- バグ修正
- UI の軽微な改善
- ドキュメント修正
- テスト追加
- ログ出力やデバッグ支援の追加
- 明確に依頼されたリファクタリング
- 保守上必要な最小限の整理

### 原則として勝手に変更しないもの
- プロジェクト全体のディレクトリ構成
- Unity のバージョン
- Package 構成
- Input System 設定
- ビルド設定
- シーン構成の大幅変更
- 既存アセットの全面差し替え
- セーブデータ仕様
- ネットワーク仕様
- 外部サービス連携設定
- チーム全体に影響する共通基盤

これらに触れる必要がある場合は、変更理由と影響範囲を明確にすること。

## 6. 開発環境共通ルール
本プロジェクトは Windows / Mac の両環境で開発を行うため、環境差異による問題を避けるために以下を守ること。

### 6.1 文字コード
- テキストファイルの文字コードは原則として UTF-8 を使用する。
- Shift-JIS など、環境依存で文字化けしやすい文字コードは使用しない。
- README、Markdown、ソースコード、設定ファイル、JSON、YAML、CSV、TSV、テキストメモなどは UTF-8 で保存する。
- 文字コードの変更だけを目的とした不要な差分は作らない。

### 6.2 改行コード
- 改行コードは原則として LF を使用する。
- OS ごとの自動変換により不要な差分が出ないよう注意する。
- 改行コードのみが変化した差分をそのままコミットしない。

### 6.3 ファイル名・ディレクトリ名
- ファイル名およびディレクトリ名は、原則として英数字を使用する。
- スペース、日本語、機種依存文字、環境依存文字はできるだけ使用しない。
- 大文字小文字の違いだけによる名前変更は避ける。
- Windows / Mac 間で扱いが不安定になる命名は避ける。

### 6.4 パスの扱い
- Windows / Mac でパス区切りや大文字小文字の扱いが異なることを意識する。
- パスの直書きは最小限にする。
- 必要に応じて共通処理や定数化を行う。
- 絶対パス依存の実装は避ける。

### 6.5 エディタ設定差異への注意
- エディタや IDE の設定差異による大量差分をそのままコミットしない。
- 保存時の自動整形や改行コード変換による差分に注意する。
- チームで使用するエディタ設定ファイルがある場合は、それに従う。

## 7. Git 運用ルール
### 基本ルール
- **`main` または `master` ブランチへ直接コミットおよび直接プッシュすることは絶対に禁止する。**
- 現在 `main` ブランチにいる場合は、**必ず作業ブランチを切ってから**開発を進めること。
- 作業完了後は、プルリクエスト等を経由して任意のタイミングで `main` ブランチへマージすること。
- 1ブランチ 1目的を原則とする。
- コミットは意味のある単位で分ける。
- 作業前に最新のブランチ状況を確認する。
- コミット前に差分を確認し、不要な変更が混ざっていないことを確認する。
- マージ前に競合の有無を確認する。
- 長期間ブランチを放置しない。

### ブランチ名
ブランチ名の詳細な規約は `Docs/01_Overview/CodingRules.md` に従うこと。  
本プロジェクトでは、用途が分かる接頭辞を付け、内容はスネークケースで記述する。

### コミットメッセージ
コミットメッセージは簡潔で内容が分かるようにする。  
先頭の種別は英語、内容説明は日本語でもよい。

例:
- `feat: プレイヤー移動処理を追加`
- `fix: タイトル画面から遷移できない問題を修正`
- `docs: セットアップ手順を更新`

## 8. Unity 開発で特に守ること
### 8.1 `.meta` ファイル
- `.meta` ファイルは Unity 管理上重要であるため、勝手に削除・再生成しない。
- アセットの移動やリネーム時は `.meta` の整合性に注意する。
- 不要に大量の `.meta` 差分を出さない。

### 8.2 Scene / Prefab
- Scene と Prefab は競合しやすいため、複数人で同時編集しそうな場合は担当を明確にする。
- 大規模な Scene 変更は、できるだけ小分けにする。
- 関係ないオブジェクトの位置変更や Inspector 差分を含めないよう注意する。
- Prefab の Apply / Revert は意図を確認してから行う。
- Scene、Prefab、ProjectSettings を変更する場合は、関係のない差分を含めないよう特に注意する。

### 8.3 ProjectSettings / Packages
- `ProjectSettings` や `Packages/manifest.json` の変更は影響が大きいため、必要性を明記する。
- 依存追加・更新は勝手に行わない。
- Unity バージョン差異による自動差分に注意する。

### 8.4 アセット
- 重いバイナリファイルの追加は最小限にする。
- 不要なアセットを大量にコミットしない。
- 仮素材、テスト素材、出力物は置き場所を分ける。
- 実運用に不要な個人用ファイルはコミットしない。

### 8.5 個人環境依存ファイル
- 個人のローカル設定やキャッシュに由来するファイルはコミットしない。
- Unity が自動生成する一時ファイルやビルド生成物は必要がない限り管理対象に含めない。
- `.gitignore` の内容を守る。

## 9. AI エージェントの行動ルール
AI エージェントは以下を守ること。

1. まず関連ファイルを確認し、既存実装を理解する。  
2. 指示された範囲に集中して修正する。  
3. 無関係な修正や過剰な最適化をしない。  
4. 人が読む説明、コメント、タスク記述は日本語で書く。  
5. コーディング規約は `Docs/01_Overview/CodingRules.md` に従う。  
6. 大きな変更を行う場合は、変更方針が分かる形で整理する。  
7. 変更後は、内容・理由・対象ファイル・確認事項・懸念点を報告する。  
8. 動作未確認なのに「問題なし」と断定しない。  
9. シークレット情報、API キー、個人情報を書き込まない。  
10. Unity 特有の差分は、意図しない変更が混ざっていないか注意する。  
11. 依頼範囲外の Inspector 調整や自動差分を勝手に確定しない。  
12. 既存コードが規約から外れていても、依頼範囲外の全面修正は行わない。  
13. 文字コードや改行コードの変化だけによる不要差分を発生させない。  
14. Windows / Mac の混在環境を前提に、環境依存の記述を避ける。

## 10. タスク対応ルール
- タスクはできるだけ小さく分割する。
- タスク名と目的は日本語で明確に書く。
- 着手時に、何を変更するかを簡潔に整理する。
- 完了時に、何を確認したかを残す。
- 未確認事項や保留事項があれば明記する。
- 長期作業になる場合は、途中経過が分かるように共有する。

### タスク記述例
- プレイヤーのジャンプ処理を追加する
- タイトル画面から設定画面への遷移を修正する
- 失敗時にエラーメッセージを表示するようにする

---

## 11. PR / レビュー方針
PR には最低限、以下を日本語で記載すること。

- 目的
- 変更内容
- 対象ファイル
- 確認内容
- 懸念点 / 未確認事項

### PR 記載例
- 目的: プレイヤー移動の基本操作を実装するため
- 変更内容: 移動入力、速度設定、アニメーション連携を追加
- 対象ファイル: `PlayerController.cs`, `PlayerAnimator.cs`
- 確認内容: 前後左右移動、停止、移動アニメーション切替を確認
- 懸念点: ゲームパッド入力は未確認

### レビュー時の観点
- 目的に対して差分が適切か
- 無関係な差分が含まれていないか
- Unity 特有の不要差分が混ざっていないか
- 文字コードや改行コードだけの差分が混ざっていないか
- コメントや説明が日本語で十分か
- 今後の保守で困らない構成か

---

## 12. 競合を減らすためのルール
- Scene、Prefab、ProjectSettings は特に競合しやすいと認識する。
- 同じファイルを複数人で触る場合は、担当範囲を先に共有する。
- 大きな変更前には対象ファイルを共有する。
- 差分が大きくなる前に早めに PR 化する。
- マージ前に競合の有無を確認し、必要に応じて最新ブランチを取り込む。
- 競合解消時は、意図しない上書きが発生していないか必ず確認する。

---

## 13. ドキュメント更新ルール
以下の場合は、必要に応じて README や関連文書も更新する。

- 導入手順が変わったとき
- 必要な Package が変わったとき
- 起動方法が変わったとき
- ビルド方法が変わったとき
- 操作方法が変わったとき
- 重要な設計判断が追加されたとき
- 環境依存の注意点が増えたとき

## 14. 禁止事項
- main への直接コミットおよび直接プッシュ
- 無関係な大量変更
- 勝手な全面リファクタリング
- 勝手な Package 追加
- 勝手な Unity バージョン変更
- `.meta` の安易な削除
- 動作未確認の断定
- 一時的なデバッグコードの放置
- 人が読む必要のある説明を英語だけで済ませること
- 目的が不明確な曖昧なブランチ作成やコミット
- Shift-JIS など環境依存文字コードの使用
- 改行コードだけが変わった不要差分のコミット
- 環境依存パスや個人用設定の安易なコミット

## 15. 変更後の報告フォーマット
作業後およびIssue完了時は、以下の形式で日本語報告すること。
また、**対応が後から確認できるように必ず `Docs/04_Logs/IssueXX_Report.md` (XXはIssue番号などの識別名) として作業ログファイルを作成し、コミット・マージすること。**

### コミュニケーションおよびログファイルへの記載内容
- 変更内容
- 変更理由
- 対象ファイル
- 確認した内容
- 未確認事項 / 懸念点

例:
- 変更内容: タイトル画面からゲーム開始時の遷移処理を追加
- 変更理由: 開始ボタンを押してもシーン遷移しなかったため
- 対象ファイル: `TitleMenu.cs`, `SceneLoader.cs`
- 確認した内容: 開始ボタン押下でゲームシーンへ移動することを確認
- 未確認事項: ビルド後の動作は未確認

## 17. ドキュメント構成・管理ルール
プロジェクトの `.md` ドキュメントは、抽象度・重要度（レベル）に応じてフォルダを階層化して管理する。
浅い階層（上位レベル）のものほどプロジェクト全体を俯瞰できる重要なドキュメントとし、深い階層（下位レベル）へ行くほど具体的で細かい設定や詳細な内容とする。
新規ドキュメントを作成・追加する際は、この階層構造に従って適切なフォルダに配置すること。

### フォルダ構成とドキュメントのインデックス
本プロジェクトでは `AGENTS.md` を起点（ハブ）とし、条件に応じて以下の各ドキュメントを参照する。

- **ルートディレクトリ (Highest Level)**
  - `README.md`: プロジェクトの顔となる最も重要な情報。
  - `AGENTS.md`: 運用ルールの中核。すべてのルールの起点となる。

- **`Docs/01_Overview/` (High Level - 全体方針・規約)**
  - 全体に適用される方針、規約、アーキテクチャ概要など俯瞰的なドキュメント。
  - 要約・詳細確認用: [`CodingRules.md`](Docs/01_Overview/CodingRules.md)

- **`Docs/02_Specifications/` (Middle Level - 仕様・設計)**
  - 各機能やモジュール単位の仕様、設計資料、挙動の詳細。

- **`Docs/03_Tasks/` (Lower Level - 作業・タスク)**
  - 進行中のタスク、Issueのまとめ、短中期的なマイルストーンなど流動的なもの。
  - 進行管理・詳細確認用: [`TASK.md`](Docs/03_Tasks/TASK.md)

- **`Docs/04_Logs/` (Lowest Level - ログ・議事録・メモ)**
  - 作業ログ、エラー調査ログ、会議議事録、手順のメモなど、記録としての要素が強いもの。

AI エージェントおよび開発メンバーは、何かについて確認・調査する際、まずはこのインデックスから適切な階層・ファイルを判断し、必要に応じて詳細なドキュメントを読み込むこと。

### 新規ルールの追加・ドキュメント分割方針
ルールや方針を追加・整理する際は、以下の手順に従うこと。
1. **既存ドキュメントの確認**: 新たなルールを追加する前に、まず既存のドキュメント（各フォルダ内の `.md` ファイル）を確認し、同種または類似カテゴリのルールがすでに存在しないか把握する。
2. **既存ドキュメントへの追記**: 類似する種類のルールを扱うドキュメントがすでに存在する場合、該当するドキュメント内に追記する（原則として同じ種類のルールは1つのファイルにまとめる）。
3. **新規ドキュメントの作成**: 既存のどのドキュメントにも該当しない「新種のルール」である場合のみ、新規で `.md` ドキュメントを作成する。
4. **適切な配置とフォルダの新設**: 新規ドキュメントを配置する際、その内容の抽象度・重要度に合わせて適切なフォルダ（`Docs/01_Overview` 〜 `Docs/04_Logs` など）に保存する。分類に適したフォルダが存在しない場合は、フォルダごと新規作成しても構わない。その際、「**階層が深くなるほど具体的で細かい設定になる**」という原則に従ってディレクトリ階層を設計すること。
5. **インデックスの更新**: 新規ドキュメントを作成した場合は、必ず本ファイル（`AGENTS.md`）のインデックス（目次）にも参照パスと用途を追記し、全体からアクセス可能な状態を保つこと。
