# チーム基本方針 (Basic Policies)

本プロジェクトにおける最も基礎的な考え方、言語の取り扱い、作業範囲、および絶対禁止事項を定めます。

## 1. 基本方針
- 変更はできるだけ小さな単位で行う。
- 1つの作業ブランチでは、1つの目的に集中する。
- 無関係な修正を同時に行わない。
- 既存の構成、命名、実装方針を尊重する。
- 大きな変更は、勝手に全面置換せず、影響範囲を明確にして進める。
- 仕様が不明確な場合は、README、既存コード、Issue、設計メモを優先して判断する。
- 判断に迷う場合は、保守的な変更を優先する。
- 複数人で同時開発していることを常に意識し、競合や手戻りを最小限にする。
- 変更前後で差分を確認し、意図しない変更が混ざっていないことを確認する。

---

## 2. 言語方針
このプロジェクトでは、人が読む必要のある情報は原則として日本語で記述する。

### 日本語で記述するもの
- タスクの説明
- ウォークスルー
- Issue の説明
- PR の説明
- 作業報告
- コード内コメント
- ドキュメント
- TODO コメント
- デバッグ手順
- 引き継ぎメモ
- レビューコメント
- 作業手順書

### 例外（英語を許容するもの）
- Unity や C# の予約語
- API 名、ライブラリ名、クラス名、メソッド名
- 一般的な技術用語として英語の方が自然なもの
- ブランチ名
- コミット種別（feat, fix, docs など）
- Unity の標準機能名、Inspector 上の正式名称

### コメント方針
- コメントは日本語で書く。
- 「何をしているか」だけでなく、「なぜ必要か」が分かるコメントを優先する。
- 一時的なメモや曖昧なコメントは残さない。
- 将来読んだ人が誤解しやすいコメントは書かない。

---

## 3. 作業範囲
### 変更してよいもの
- 依頼された機能追加
- バグ修正
- UI の軽微な改善
- ドキュメント修正
- テスト追加
- ログ出力やデバッグ支援の追加
- 明確に依頼されたリファクタリング
- 保守上必要な最小限の整理

### 原則として勝手に変更しないもの
- プロジェクト全体のディレクトリ構成
- Unity のバージョン
- Package 構成
- Input System 設定
- ビルド設定
- シーン構成の大幅変更
- 既存アセットの全面差し替え
- セーブデータ仕様
- ネットワーク仕様
- 外部サービス連携設定
- チーム全体に影響する共通基盤

これらに触れる必要がある場合は、変更理由と影響範囲を明確にすること。

---

## 4. 禁止事項
以下の行為はチーム開発を阻害するため、絶対に行わないこと。

- main への直接コミットおよび直接プッシュ
- 無関係な大量変更
- 勝手な全面リファクタリング
- 勝手な Package 追加
- 勝手な Unity バージョン変更
- `.meta` の安易な削除
- 動作未確認の断定
- 一時的なデバッグコードの放置
- 人が読む必要のある説明を英語だけで済ませること
- 目的が不明確な曖昧なブランチ作成やコミット
- Shift-JIS など環境依存文字コードの使用
- 改行コードだけが変わった不要差分のコミット
- 環境依存パスや個人用設定の安易なコミット
