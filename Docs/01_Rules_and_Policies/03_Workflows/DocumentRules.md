# ドキュメント管理・更新ルール (Document Rules)

プロジェクト内のドキュメント（Markdownファイル等）の更新タイミングと、ディレクトリの構成・管理に関するルールです。

## 1. ドキュメント更新ルール
以下の場合は、必要に応じて README や関連文書も更新する。

- 導入手順が変わったとき
- 必要な Package が変わったとき
- 起動方法が変わったとき
- ビルド方法が変わったとき
- 操作方法が変わったとき
- 重要な設計判断が追加されたとき
- 環境依存の注意点が増えたとき

---

## 2. ドキュメント構成・管理ルール
プロジェクトの `.md` ドキュメントは、抽象度・重要度（レベル）に応じてフォルダを階層化して管理する。
浅い階層（上位レベル）のものほどプロジェクト全体を俯瞰できる重要なドキュメントとし、深い階層（下位レベル）へ行くほど具体的で細かい設定や詳細な内容とする。
新規ドキュメントを作成・追加する際は、原則として必ず `Docs` ディレクトリ以下の階層構造に従って適切なフォルダに配置すること。ただし、ルート直下の `README.md` および `AGENTS.md` は例外とする（これら以外をルートや別ディレクトリに作成しないこと）。

### 2.1 フォルダ構成とドキュメントのインデックス
本プロジェクトでは `AGENTS.md` を起点（ハブ）とし、具体的なルールは `Docs` 以下を参照する。

- **ルートディレクトリ (Highest Level)**
  - `README.md`: プロジェクトの顔となる最も重要な情報。
  - `AGENTS.md`: ドキュメントルートおよび主要なルールのポータル。

- **`Docs/01_Rules_and_Policies/` (High Level)**
  - 全体に適用される方針、規約、開発環境、Gitワークフローなどのルール。さらに下層のカテゴリフォルダに分かれる。

- **`Docs/02_Specifications/` (Middle Level)**
  - 各機能やモジュール単位の仕様、設計資料、挙動の詳細。

- **`Docs/03_Tasks/` (Lower Level)**
  - 進行中のタスク、Issueのまとめ、短中期的なマイルストーンなど流動的なもの。

- **`Docs/04_Logs_and_Reports/` (Lowest Level)**
  - 作業ログ、エラー調査ログ、会議議事録、事後報告のメモなど、記録としての要素が強いもの。

### 2.2 新規ルールの追加・ドキュメント分割方針
ルールや方針を追加・整理する際は、以下の手順に従うこと。
1. **既存ドキュメントの確認**: 新たなルールを追加する前に、まず既存のドキュメントを確認し、同種または類似カテゴリのルールがすでに存在しないか把握する。
2. **既存ドキュメントへの追記**: 類似する種類のルールを扱うドキュメントがすでに存在する場合、該当するドキュメント内に追記する（原則として同じ種類のルールは1つのファイルにまとめる）。
3. **新規ドキュメントの作成**: 既存のどのドキュメントにも該当しない「新種のルール」である場合のみ、新規で `.md` ドキュメントを作成する。
4. **適切な配置とフォルダの新設**: 新規ドキュメントを配置する際、その内容の抽象度・重要度に合わせて適切なフォルダ（`Docs/01_Rules_and_Policies` 等のサブフォルダ）に保存する。分類に適したフォルダが存在しない場合は、フォルダごと新規作成しても構わない。その際、「**階層が深くなるほど具体的で細かい設定になる**」という原則に従ってディレクトリ階層を設計すること。
5. **インデックスの更新**: 新規ドキュメントを作成した場合は、必要に応じて `Overview.md` 等のインデックス（目次）にも参照パスと用途を追記し、アクセス可能な状態を保つこと。
