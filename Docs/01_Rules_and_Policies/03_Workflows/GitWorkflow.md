# Git ブランチ・PR・競合運用のワークフロー

このプロジェクトでは、以下の原則をシステムの絶対ルールとして遵守します。

## 1. 【絶対禁止事項】
**`main` または `master` ブランチへ直接コミットおよび直接プッシュすることは絶対に禁止する。**

## 2. 作業開始時のブランチ作成
新しいIssueやタスクに取り組む際は、**必ず** `main` ブランチの最新状態から新しい作業用ブランチを作成して Checkout すること。無関係な修正は同時に行わず、1ブランチ 1目的を原則とする。
長期間ブランチを放置しないこと。

```bash
git checkout main
git pull origin main
git checkout -b feature/issueX_description
```

### 2.1 ブランチ名の命名規則
用途が分かる接頭辞を付け、内容はスネークケースで記述する。
（※詳細は `CodingRules.md` にて規定されている通り、`feature/`, `fix/`, `hotfix/` などを使用）

## 3. 作業中のコミットとプッシュ
作業中は、切ったブランチ（`feature/...`など）に対してコミットを積み重ね、リモートの同ブランチへそのままプッシュする。コミットは意味のある単位で分け、コミット前に差分を確認して不要な変更が混ざっていないことを確認する。

### 3.1 コミットメッセージ
コミットメッセージは簡潔で内容が分かるようにする。
先頭の種別は英語、内容説明は日本語でもよい。
- `feat: プレイヤー移動処理を追加`
- `fix: タイトル画面から遷移できない問題を修正`
- `docs: セットアップ手順を更新`

```bash
git add .
git commit -m "feat: xxx"
git push -u origin feature/issueX_description
```

### 3.2 巨大な変更や大量のアセット追加時の対応
一度のコミット・プッシュでファイルサイズが巨大（数百MB～GB単位）になる場合、GitHubの制限（1ファイル100MBまで）や通信タイムアウト（HTTP 500エラー等）でプッシュが失敗することがあります。
そのため、大量のアセットや重いファイルを追加する際は、以下のルールに従ってください。
- **一度にすべてをコミット・プッシュせず、ディレクトリ単位やカテゴリ単位などで複数回に分割してコミットおよびプッシュを行うこと。**
- `*.unitypackage` など、リポジトリに含める必要のない巨大なインストーラーファイルは原則としてコミットせず、必要に応じて `.gitignore` で除外すること。

## 4. PR (Pull Request) / レビュー方針
差分が大きくなる前に早めに PR 化する。PR には最低限、以下を日本語で記載すること。

- 目的
- 変更内容
- 対象ファイル
- 確認内容
- 懸念点 / 未確認事項

### 4.1 レビュー時の観点
- 目的に対して差分が適切か
- 無関係な差分が含まれていないか
- Unity 特有の不要差分が混ざっていないか
- 文字コードや改行コードだけの差分が混ざっていないか
- コメントや説明が日本語で十分か
- 今後の保守で困らない構成か

## 5. 【重要】AIエージェントによるマージの禁止
作業が完了し動作確認が終わっても、**AIエージェントが自己判断で `git checkout main && git merge feature/...` を実行してはいけません。**
直線的な履歴にならないようにするためにも、AIのロールは「作業ブランチをプッシュして報告する」ところまでです。

## 6. ユーザーへの報告と承認待ち（AI向け手順）
ブランチをプッシュしたら、ユーザーに「Pushが完了したこと」「内容の確認をお願いしたいこと」を通知（`notify_user`）し、待機してください。
ユーザーから「マージして下さい」「次のIssueに行っていいよ（=マージ承認と同義）」などの**明確な許可が出た場合にのみ**、初めて `main` へのマージを行ってください。

## 7. マージ時の手順とオプション（競合の解消）

### 7.1 競合を減らすためのルール
- Scene、Prefab、ProjectSettings は特に競合しやすいと認識する。
- 同じファイルを複数人で触る場合は、担当範囲を先に共有する。
- マージ前に競合の有無を確認し、必要に応じて最新ブランチを取り込む。
- 競合解消時は、意図しない上書きが発生していないか必ず確認する。

### 7.2 【重要】マージ時の `--no-ff` オプションの徹底
作業ブランチを `main` へ合流させる際、これまでの直線的な履歴（Fast-Forward）になるのを防ぎ、作業の分岐と合流の線を明確に残すために**必ず `--no-ff` オプションを付けてマージ**してください。

```bash
git checkout main
git merge --no-ff feature/issueX_description -m "Merge branch 'feature/issueX_description'"
git push origin main
```
